Guia de Boas Práticas — Etapa 1

🧩 ETAPA 1: DEFINIR

🎯 Objetivo da Etapa: Estabelecer, antes do início do desenvolvimento, todos os requisitos mínimos de segurança necessários para garantir que o sistema não seja construído com vulnerabilidades estruturais, reduzindo riscos técnicos, operacionais e legais.

🧠 1. COMPREENSÃO DO CONTEXTO DO SISTEMA

Antes de qualquer decisão técnica, o desenvolvedor deve compreender claramente o contexto do sistema. Isso inclui a definição do seu objetivo, o problema que resolve, o perfil dos usuários e o ambiente em que será executado. Sistemas expostos à internet devem ser tratados como ambientes hostis por padrão, exigindo maior rigor na aplicação de controles de segurança.

👉 Boa prática: Nunca iniciar o desenvolvimento sem compreender o nível de criticidade e exposição do sistema.

🧠 2. CLASSIFICAÇÃO E TRATAMENTO DE DADOS

Identificar e classificar todos os dados que serão manipulados, com atenção especial aos dados sensíveis (credenciais, informações pessoais e financeiras). Essa classificação define diretamente os requisitos de proteção, como uso de criptografia, controle de acesso e retenção.

👉 Boa prática: Dados sensíveis nunca devem ser armazenados ou transmitidos sem proteção adequada.

🧠 3. DEFINIÇÃO DE AUTENTICAÇÃO

A autenticação deve ser definida de forma estruturada. Estabelecer políticas de senha, critérios de complexidade, mecanismos de recuperação de acesso e controle de sessão. Credenciais devem ser armazenadas utilizando algoritmos de hash seguros.

👉 Boa prática: A autenticação deve ser projetada antes da implementação, nunca improvisada durante o desenvolvimento.

🧠 4. CONTROLE DE ACESSO (AUTORIZAÇÃO)

Definir claramente os diferentes perfis de usuários e permissões. O princípio do menor privilégio deve ser aplicado. O controle de acesso deve ser validado no backend, independentemente das restrições implementadas na interface.

👉 Boa prática: Nunca confiar no frontend para controle de permissões.

🧠 5. VALIDAÇÃO DE ENTRADAS E SUPERFÍCIE

Tratar entradas como potencialmente maliciosas. Definir previamente estratégias de validação, sanitização e restrição de dados, incluindo limites de tamanho, formatos esperados e controle de uploads para prevenir injeção de código e scripts maliciosos.

👉 Boa prática: Toda entrada deve ser validada no backend antes de qualquer processamento.

🧠 6. DEFINIÇÃO DA ARQUITETURA E EXPOSIÇÃO

Definir onde e como o sistema será executado, incluindo ambiente de hospedagem, serviços externos e APIs. Cada ponto de comunicação representa uma possível superfície de ataque e deve ser considerado no planejamento.

👉 Boa prática: Minimizar a superfície de ataque desde a arquitetura.

🧠 7. GESTÃO DE DEPENDÊNCIAS

O uso de bibliotecas e frameworks deve ser controlado e monitorado. Selecionar dependências confiáveis, mantê-las atualizadas e evitar o uso de código de origem desconhecida sem validação rigorosa.

👉 Boa prática: Toda dependência deve ser tratada como potencial risco.

🧠 8. CONFIGURAÇÃO SEGURA DO AMBIENTE

Ambientes de dev, teste e prod devem ser separados. O sistema nunca deve ser executado em produção com configurações de debug ativas. Credenciais não devem estar expostas no código-fonte.

👉 Boa prática: Erros de configuração são uma das principais causas de vulnerabilidades.

🧠 9. MONITORAMENTO E REGISTRO

Definir previamente quais eventos serão registrados pelo sistema, como tentativas de login, falhas de autenticação e ações críticas. Esses registros permitem identificar comportamentos suspeitos.

👉 Boa prática: Se não há monitoramento, não há controle de segurança.

🧠 10. CONTINUIDADE E RECUPERAÇÃO

O sistema deve possuir estratégia de backup e recuperação, garantindo que dados possam ser restaurados em caso de falha ou incidente. O tempo tolerável de indisponibilidade deve ser predefinido.

👉 Boa prática: Segurança também envolve garantir disponibilidade.

✅ AUTOAVALIAÇÃO DO DESENVOLVEDOR

🔹 1. SOBRE OS DADOS

Os dados manipulados foram classificados (sensíveis, pessoais, comuns)?
Dados sensíveis estão protegidos (criptografia ou hash)?
Existe justificativa para coleta de cada dado?
Existe definição de onde os dados serão armazenados?
Existe definição de tempo de retenção dos dados?

🔹 2. SOBRE AUTENTICAÇÃO

Senhas são armazenadas com hash seguro (bcrypt/argon2)?
Existe política mínima de senha (tamanho/complexidade)?
Sessões possuem expiração automática?
Existe proteção contra múltiplas tentativas de login?
Existe mecanismo seguro de recuperação de senha?

🔹 3. SOBRE CONTROLE DE ACESSO

Existem perfis de usuário definidos?
O sistema aplica princípio do menor privilégio?
Permissões são validadas no backend?
Existe proteção contra acesso direto por URL/API?

🔹 4. SOBRE ENTRADAS E VALIDAÇÃO

Todas as entradas são validadas no backend?
Existe proteção contra SQL Injection (queries parametrizadas)?
Existe proteção contra XSS?
Existe controle de tamanho/formato de entrada?

🔹 5. SOBRE ARQUITETURA E EXPOSIÇÃO

O sistema utiliza HTTPS obrigatoriamente?
Serviços desnecessários estão desativados?
Integrações externas são validadas e seguras?

🔹 6. SOBRE DEPENDÊNCIAS

Bibliotecas utilizadas são confiáveis e atualizadas?
Existe controle sobre dependências utilizadas?

🔹 7. SOBRE CONFIGURAÇÃO

Credenciais estão fora do código (env)?
Ambiente de produção está sem debug ativo?

🔹 8. SOBRE MONITORAMENTO

O sistema registra eventos críticos (logins, erros)?
Logs evitam exposição de dados sensíveis?

🔹 9. SOBRE RESILIÊNCIA

Existe estratégia de backup definida?
